STEP 2 — Drain the drive_push_queue
==============================================================================
Fetch /api/scheduled/drive-push/pending. For each row:

  a. Determine target Drive folder:
       - Use row.canonicalParentFolderId as the parent (already provided).
       - Use row.subfolderName as the subfolder name under that parent.
       - If row.targetSubpath is non-empty (e.g. "2026-05"), nest one more level inside the subfolder. Create month folders on demand.
       - Final path: HubRoot / {canonicalParent} / {subfolderName} / {targetSubpath?} / {fileName}

  b. Resolve file content:
       - If row.contentText is non-empty AND row.mimeType starts with "text/", upload the text directly with the given mimeType.
       - Otherwise download row.fileUrl (it returns a redirect to S3) and upload the bytes.

  c. DEDUPE-ON-UPLOAD:
       - Before creating, search the target folder for a file with the same name.
       - If a file with the same name already exists, UPDATE that file's content (do not create a duplicate); use its existing driveFileId.

  d. POST { id, status: "pushed", driveFileId } back to /api/scheduled/drive-push/result.
     On failure, POST { id, status: "failed", errorMessage }.
     If the row's target_folder is one this playbook does not recognize, POST status: "skipped" with errorMessage="unhandled target" — never leave a row pending forever.
